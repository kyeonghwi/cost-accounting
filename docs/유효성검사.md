# 유효성 검사 레이어 — lib/zod/

## 왜 숫자(number)가 아닌 문자열(string)로 십진수를 표현하는가

Prisma는 `Decimal(18,4)` 컬럼을 `decimal.js` 라이브러리의 `Decimal` 타입으로 매핑한다. JavaScript의 `number` 타입(IEEE 754 배정밀도)은 18자리 정수 전체나 소수점 이하 4자리 값을 정밀도 손실 없이 표현하지 못한다. 예를 들어 JS에서 `0.1 + 0.2 === 0.30000000000000004`가 된다.

API 경계에서 `/^\d+(\.\d{1,4})?$/` 정규식에 맞는 문자열만 허용하면:
- Prisma에 도달하기 전에 부동소수점 반올림 오류가 시스템에 유입되지 않는다
- 정규식이 진입점에서 소수점 이하 4자리 계약을 강제한다
- 호출자는 제출 전에 숫자를 문자열로 변환(`amount.toFixed(4)`)해야 한다

## z.infer 패턴

각 스키마는 스키마 정의에서 직접 파생된 TypeScript 타입을 내보낸다:

```typescript
export const CostEntryCreateSchema = z.object({ ... })
export type CostEntryCreateInput = z.infer<typeof CostEntryCreateSchema>
```

`CostEntryCreateInput`은 `safeParse`가 성공 시 반환하는 결과와 구조적으로 동일하다. 서버 액션은 유효성 검사 후 이 타입을 받는다 — 별도의 DTO 타입이 필요 없다.

## 예시: CostEntryCreateSchema → CostEntryCreateInput

```typescript
import { CostEntryCreateSchema, CostEntryCreateInput } from '@/lib/zod/costEntry'

async function createCostEntry(raw: unknown) {
  const parsed = CostEntryCreateSchema.safeParse(raw)
  if (!parsed.success) {
    return { error: parsed.error.flatten() }
  }
  const data: CostEntryCreateInput = parsed.data
  // data.hours는 숫자 8이 아닌 문자열 "8.0000"
  return prisma.costEntry.create({ data })
}
```

## 스키마 사용처

| 스키마 파일 | 사용하는 곳 |
|-------------|-------------|
| `lib/zod/costEntry.ts` | `app/(master)/cost-entries/actions.ts` |
| `lib/zod/allocation.ts` | `app/(close)/actions.ts` (실행 및 규칙 관리) |
| `lib/zod/transfer.ts` | `app/(master)/transfer-markup/actions.ts` |
| `lib/zod/master.ts` | `app/(master)/*/actions.ts` (조직, 프로젝트, 인원 등) |
| `lib/zod/period.ts` | `app/(close)/actions.ts` (기간 마감 게이트) |

T7/T10에서 연결 예정. 각 서버 액션은 위에 나온 `safeParse → 조기 오류 반환 → prisma 호출` 패턴을 따른다.

## ID 필드 참고

Prisma는 모든 `@id` 필드에 `cuid()`를 사용한다. CUID는 `c`로 시작하며 약 25자다. `idField` 헬퍼는 `z.string().uuid()` 대신 `z.string().min(1)`을 사용하여, 향후 UUID 마이그레이션 시 유효성 검사 레이어를 변경하지 않고 CUID와 UUID를 모두 허용한다.
