## 3. 배부 알고리즘 상세

**소스 파일**: `lib/allocation/`

### 3-1. 배부 드라이버

| 드라이버 키 | 설명 | 기준 |
|------------|------|------|
| `HEADCOUNT` | 수령 부서의 인원 수 | 기간 시작 시점 인원 |
| `DIRECT_LABOR_HOURS` | 수령 부서의 직접 노동시간 | 기간 실제 DLH |
| `DIRECT_COST` | 수령 부서의 직접원가 | 기간 실제 원가 |
| 사용자 정의 | 풀의 `customKey` 필드 설정 | 직접 입력 수치 |

### 3-2. 배부 기본 공식

풀 잔액 `B`, 수령 대상 집합에 대해:

```
드라이버 합계 = Σ driver[i]   (모든 수령 대상 i에 대해)
배부 비율[i]  = driver[i] / 드라이버 합계
배부액[i]     = B × 배부 비율[i]
```

### 3-3. 직접배부법 (Direct Allocation)

각 서비스 부서 원가를 생산 부서에 직접 배부한다. 서비스 부서 간 서비스 제공을 무시한다.

**예시**: IT 간접비 ₩120,000, 드라이버 = 인원 수

| 부서 | 인원 | 배부 비율 | 배부액 |
|------|------|---------|--------|
| 기계 부서 | 40명 | 40% | ₩48,000 |
| 조립 부서 | 60명 | 60% | ₩72,000 |

### 3-4. 단계배부법 (Step-Down Allocation)

서비스 부서를 우선순위 순서대로 배부한다. 이미 배부가 완료된 부서는 이후 배부에서 수령 대상에서 제외된다.

```
k번째 배부 단계 (D_k가 배부 중):
  수령 집합 = { 전체 대상 } - { D_1, D_2, ..., D_{k-1} }
  배부 비율[i] = driver[i] / Σ driver[j] (j가 수령 집합에 속할 때)
  배부액[i] = D_k 잔액 × 배부 비율[i]
```

### 3-5. 잔차 조정 (Rounding Adjustment)

배부 비율이 균등하게 나뉘지 않을 때 부동소수점 잔차가 발생한다. 마지막 수령 대상에 잔차를 추가하여 `Σ 배부액 = 풀 잔액`이 항상 성립하도록 보정한다.

```
잔차 = B - Σ 배부액[i]   (마지막 수령 대상 제외)
마지막 배부액 += 잔차
```

허용 오차: `|잔차| > 0.005 × B`이면 오류 발생 (조용한 오류 흡수 방지).

### 3-6. 정확성 보장 장치

- **ConservativeAdd** (`lib/allocation/decimal.ts`): `Σ 배부액 = 풀 잔액` 불변식을 마지막 단계 조정으로 보장
- **SHA-256 체크섬** (`lib/allocation/checksum.ts`): 배부 입력·결과를 해시화하여 `AllocationRun`에 저장. 재실행 시 동일 입력 = 동일 해시 (결정론적 보장)
- **Decimal 산술** (`lib/allocation/decimal.ts`): `Decimal.precision=28`, `Money` 서브클래스로 금액 단위 타입 강제

## 4. 차이분석 알고리즘 상세

**소스 파일**: `lib/variance/decompose.ts`

Kaplan/Atkinson 4분법으로 총 차이를 4개 요인으로 분해한다.

### 4-1. 기호 정의

| 기호 | 의미 |
|------|------|
| AP | 실제 단가 (Actual Price) |
| SP | 표준 단가 (Standard Price / 예산 단가) |
| AV | 실제 수량 (Actual Volume) |
| BV | 예산 수량 (Budgeted Volume) |
| AM | 실제 제품 믹스 비율 (단일 제품 = 1.0) |
| BM | 예산 제품 믹스 비율 (단일 제품 = 1.0) |
| AC | 실제 단위원가 |
| SC | 표준 단위원가 |

### 4-2. 4개 요인 수식

**가격 차이 (Price Effect)**
```
priceEffect = (AP - SP) × AV
```
해석: 실제 판매한 수량에 대해 표준 단가 대신 실제 단가를 받은 효과

**수량 차이 (Volume Effect)**
```
volumeEffect = (AV - BV) × SP
```
해석: 예산 대비 실제 판매량 차이가 표준 단가로 환산한 수익에 미치는 효과

**믹스 차이 (Mix Effect)**
```
mixEffect = (AM - BM) × AV × SP
```
해석: 예산과 다른 제품 구성 비율이 수익에 미치는 효과 (단일 제품 시 = 0)

**효율 차이 (Efficiency Effect)**
```
efficiencyEffect = (SC - AC) × AV
```
해석: 단위당 실제 원가가 표준 원가와 다른 것이 이익에 미치는 효과

### 4-3. 합산 및 잔차 검증

```
총 차이 = priceEffect + volumeEffect + mixEffect + efficiencyEffect
잔차 = 0  (선형 모델에서 교차 효과 미포함)
```

검증 조건: `|잔차| / |총 차이| ≤ 0.001 (0.1%)`

### 4-4. 부호 규칙

- 양수 = 유리한 차이 (실제가 표준/예산보다 좋음)
- 음수 = 불리한 차이 (실제가 표준/예산보다 나쁨)

### 4-5. 계산 예시

| 항목 | 예산/표준 | 실제 |
|------|---------|------|
| 수량 | 100 | 120 |
| 단가 | 100 | 110 |
| 단위원가 | 60 | 70 |
| 믹스 비율 | 0.50 | 0.60 |

```
가격 차이    = (110 - 100) × 120     =  1,200  유리
수량 차이    = (120 - 100) × 100     =  2,000  유리
믹스 차이    = (0.60 - 0.50) × 120 × 100 = 1,200  유리
효율 차이    = (60 - 70) × 120       = -1,200  불리
총 차이      = 1,200 + 2,000 + 1,200 - 1,200 = 3,200
잔차         = 0
```

## 5. 이전가격 엔진 상세

**소스 파일**: `lib/transfer/`

### 5-1. 이전가격 계산

```
이전가격 = 표준 요율 × 노동시간 × (1 + 마크업 비율)
이전 수익 = 공급 조직의 이전가격 합계  (TransferCredit)
이전 비용 = 수령 조직의 이전가격 합계  (TransferCharge)
```

### 5-2. 수익성 반영

이전가격을 통해 내부 서비스 부서도 수익성 지표를 가질 수 있다. 마크업을 통해 조직별 기여도를 평가한다.

## 6. 월 마감 워크플로우 상세

**소스 파일**: `lib/close/workflow.ts`

월 마감(`runClose`)은 배부 → 이전가격 → 차이분석을 단일 데이터베이스 트랜잭션으로 실행한다.

```
runClose(periodId)
    │
    ├─ 1. Period 조회 및 OPEN 상태 확인
    │
    ├─ $transaction 시작
    │    ├─ 2. AllocationRule 조회 → 배부 실행
    │    │      → AllocationResult INSERT (풀별 배부 결과)
    │    │      → AllocationRun INSERT (SHA-256 체크섬 감사)
    │    │
    │    ├─ 3. TransferMarkup 조회 → 이전가격 계산
    │    │      → TransferEntry INSERT (공급/수령 조직별)
    │    │
    │    ├─ 4. generateVarianceSnapshots()
    │    │      → VarianceSnapshot INSERT (HQ별 + 전사)
    │    │      직전 CLOSED 기간 없으면 0 기준으로 계산
    │    │
    │    └─ 5. Period.status = CLOSED 업데이트
    │
    └─ COMMIT → CloseResult 반환 (snapshotCount 포함)
```

**원자성 보장**: 5단계 중 하나라도 실패하면 전체 롤백. 부분적 마감 상태 불가.

## 7. 검증 레이어 (Zod)

**소스 파일**: `lib/zod/`

API 경계에서 모든 입력값을 Zod로 검증한다.

### 7-1. 금액 필드를 문자열로 받는 이유

JavaScript `number` 타입(IEEE 754)은 18자리 정수나 소수 4자리 값을 정확하게 표현하지 못한다. 예: `0.1 + 0.2 = 0.30000000000000004`.

API 경계에서 `/^\d+(\.\d{1,4})?$/` 정규식으로 문자열만 허용한다:
- 부동소수점 오류가 Prisma 이전에 시스템에 진입하지 않음
- 4자리 소수점 계약을 입력 시점에 강제
- 호출자는 `amount.toFixed(4)` 후 전송

### 7-2. z.infer 패턴

```typescript
export const CostEntryCreateSchema = z.object({ ... })
export type CostEntryCreateInput = z.infer<typeof CostEntryCreateSchema>
```

Server Action이 이 타입을 직접 사용 — 별도 DTO 클래스 불필요.

### 7-3. Server Action 패턴

```typescript
async function createCostEntry(raw: unknown) {
  const parsed = CostEntryCreateSchema.safeParse(raw)
  if (!parsed.success) return { error: parsed.error.flatten() }
  const data: CostEntryCreateInput = parsed.data
  // data.hours는 숫자 8이 아닌 문자열 "8.0000"
  return prisma.costEntry.create({ data })
}
```

## 9. 핵심 설계 결정

### 9-1. Decimal 산술

모든 금액 계산에 `Decimal` 타입을 end-to-end로 사용한다. `lib/allocation/decimal.ts`에서 `Decimal.precision=28`로 설정하고 `Money` 서브클래스를 통해 타입 강제.

### 9-2. 엔진 계층 격리

`lib/engines/` (구 구조) / `lib/allocation/`, `lib/variance/`, `lib/transfer/` (현재)는 Next.js import가 없다. 테스트 실행 시 Next.js 서버 없이 Vitest만으로 실행 가능.

### 9-3. Period 상태 머신

```
OPEN → (runClose 실행) → CLOSED
```
CLOSED 기간은 재마감 불가. runClose가 OPEN 상태를 먼저 확인한다.

### 9-4. 파일 크기 제한

300줄 하드 한계 (목표 200줄 이하). 소스 파일 크기 제한으로 단일 책임 원칙 강제.

    └── e2e/                # Playwright E2E (4개)

## 11. 로컬 실행 방법

```bash
# 1. 의존성 설치
pnpm install

# 2. 환경 변수 설정
cp .env.example .env.local
# DATABASE_URL="postgresql://user:password@localhost:5432/cost_accounting"

# 3. 데이터베이스 마이그레이션
pnpm db:migrate

# 4. 시드 데이터 로드 (소규모: 1 HQ, 3 프로젝트, 10 인원, 1개월)
pnpm db:seed:small
# 또는 대규모: pnpm db:seed:full

# 5. 개발 서버 시작
pnpm dev
# → http://localhost:3000

# 테스트 실행
pnpm test              # 단위 테스트
pnpm test:coverage     # V8 커버리지 포함
pnpm test:e2e          # Playwright E2E
```
